# Kernel v3.0 업그레이드 로그: Session Orchestrator

## 1. 개요 (Overview)
이 문서는 **Kernel v2.1 (감사 중심)**에서 **Kernel v3.0 (세션 오케스트레이션)**으로의 진화 과정을 기록합니다.
이번 업그레이드는 개별적으로 동작하던 역할(Role)들을 **"스쿼드(Squad)"** 시스템으로 묶고, **"세션 프로토콜(Brief → Artifact → Verdict)"**을 도입하여 유기적인 협업 체계를 구축하는 것을 목표로 합니다.

---

## 2. Core Protocol: THE SESSION LOOP (세션 프로토콜)

### 🔄 비교: v2.1 vs v3.0

| 기능 (Feature)    | v2.1 (개별 전문가)            | v3.0 (오케스트레이션 스쿼드)                     | 핵심 변경 (Key Change)       |
| :---------------- | :---------------------------- | :----------------------------------------------- | :--------------------------- |
| **운영 방식**     | 사용자가 각 Role을 개별 호출. | Role이 서로에게 **티켓(Brief)**을 발행하고 이관. | `Handoffs` 패턴 도입         |
| **입력 (Input)**  | 자연어 요청 (채팅).           | 구조화된 **세션 브리프 (Session Brief)**.        | 모호성 제거 (Zero Ambiguity) |
| **출력 (Output)** | 설명 + 코드 혼합.             | 검증 가능한 **아티팩트 (Artifact)** 단독 출력.   | `No Fluff` 정책 (잡담 금지)  |
| **종료 조건**     | 사용자 만족 시 종료.          | 리뷰어의 **판결 (Verdict)** 승인 시 종료.        | `Definition of Done` 강제    |

---

## 3. Role 01 & 02: ARCHITECT & BUILDER (발주와 수행)

### 🔄 비교: v2.1 vs v3.0

| 역할 (Role)      | v2.1 (기획과 시공)                 | v3.0 (발주와 납품)                                | 업그레이드 내용                                          |
| :--------------- | :--------------------------------- | :------------------------------------------------ | :------------------------------------------------------- |
| **01_ARCHITECT** | 계획(`todo.md`)을 수립하고 나열함. | 작업 지시서(**Session Brief**)를 발행하고 승인함. | 단순 계획자가 아닌 **권한자(Authorizer)**로 격상.        |
| **02_BUILDER**   | 요청받은 코드를 작성함.            | 지시서를 승인하고 **Build Artifact**를 제출함.    | 지시서에 없는 내용은 **거부(Reject)**하거나 되묻기 가능. |

### 🚀 핵심 업그레이드
1.  **명시적 계약 (Explicit Contracts):** 아키텍트는 더 이상 모호한 계획을 세우지 않습니다. 개발자가 *무엇을 건드려야 하고(Must Touch)*, *무엇을 건드리지 말아야 하는지(Must NOT Touch)* 명확히 지시합니다.
2.  **아티팩트 중심 (Artifact-Centric):** 빌더는 대화형 설명 대신, 리뷰어가 즉시 검수할 수 있는 포맷의 결과물만 산출합니다.

---

## 4. Role 05 & 10: SPECIALISTS (전문화와 신설)

### 🔄 비교: v2.1 vs v3.0

| 역할 (Role)          | v2.1 (지원 업무)                | v3.0 (고도화된 전문성)                          | 비고                                      |
| :------------------- | :------------------------------ | :---------------------------------------------- | :---------------------------------------- |
| **05_COMMUNICATOR**  | 설명, 교육, 문서화를 모두 담당. | **기술 문서(Docs)**와 **커밋 메시지**에만 집중. | 교육 기능은 08/09번으로 이관 (Role Diet). |
| **10_TEST_ENGINEER** | (없음 - Builder가 겸직)         | **TDD 전략 수립** 및 **E2E 테스트** 전담.       | **신규 Role 추가 (QA Specialist).**       |

### 🚀 핵심 업그레이드
1.  **No Fluff (군더더기 제거):** 05번 커뮤니케이터는 이제 불필요한 대화(Small talk)를 하지 않습니다. 오직 "진실된 기록(Documentation Artifact)"만 남깁니다.
2.  **The Beyoncé Rule:** 신설된 10번 테스트 엔지니어는 "테스트가 없으면 존재하지 않는 것"이라는 원칙하에, 코드 작성 전후로 품질을 동적으로 검증합니다.

---

## 5. 결론 (Conclusion)
* **v2.0**은 시스템에 **"두뇌(Brain)"**를 달았습니다. (추론)
* **v2.1**은 시스템에 **"교과서(Manuals)"**를 쥐여주었습니다. (감사)
* **v3.0**은 시스템을 **"하나의 회사(Company)"**로 만들었습니다. (협업)
    * 이제 각 역할은 서로 **문서(Brief)**로 지시하고, **결과물(Artifact)**로 답하며, **판결(Verdict)**로 마무리를 짓습니다.
